## Overview

v1.1 closes the PRD's full v1 scope (all payload types + pin/burn modes) and de-risks the protocol over real Mainline DHT — so cipherpost is no longer just a walking skeleton under MockTransport. The five phases continue from v1.0's Phase 4 with no phase-number reset (5–9), per coarse-granularity rule held from v1.0 retrospective.

Walking skeleton (v1.0) shipped a working end-to-end round trip under MockTransport. v1.1 adds:

- **Non-interactive automation** so scripted send/receive works without a TTY (Phase 5)
- **Three typed payload variants** (X.509 cert, PGP key, SSH key) with pattern-establish-then-apply (Phases 6–7)
- **`--pin` and `--burn` encryption modes** orthogonal to all four Material variants (Phase 8)
- **Real-DHT release-acceptance gate** + concurrent-publish CAS contract (Phase 9)

**Details:**

PIN crypto stack is **cclink-fork-with-divergence**: Argon2id(PIN + 32-byte salt) → HKDF-SHA256 `cipherpost/v1/pin` → 32-byte X25519 scalar → wrapped into age `Identity` for nested-age inner layer. **No direct `chacha20poly1305` calls** — CLAUDE.md's load-bearing rule preserved (PIN crypto stays inside `age` for AEAD). PIN salt embedded in blob (`base64(salt || age_ciphertext)`), authenticated by outer signature. `OuterRecord` gains `pin_required: bool`; v1.0 byte-identity preserved via `#[serde(default, skip_serializing_if = "is_false")]`.

`cipherpost send --burn` ships full receive-side burn semantics: `Envelope.burn_after_read=true` (inner-signed, post-decrypt; NOT OuterRecord per ciphertext-only-on-wire principle). First receive emit-before-mark order (D-P8-12) — emit decrypted bytes → fsync stdout → append `state: "burned"` ledger row → fsync ledger → touch sentinel; second receive returns exit 7 "share already consumed". Receipt published unconditionally on burn-receive (no `if !envelope.burn_after_read` guard — burn does NOT suppress attestation per BURN-04). Acceptance banner shows `[BURN — you will only see this once]` marker (literal em-dash) at TOP of header. PIN prompt rendered to stderr BEFORE typed-z32 acceptance prompt (D-P8-07/08).

Compose orthogonality: pin × burn × {GenericSecret, X509Cert, PgpKey, SshKey} 23-test matrix in `tests/pin_burn_compose.rs`. Wire-byte preservation: v1.0 fixtures byte-identical via `is_false` skip-serializing-if; new fixtures `outer_record_pin_required_signable.bin` (212 B) + `envelope_burn_signable.bin` (142 B) committed. Error-oracle hygiene extended: wrong-PIN folds into existing `Error::DecryptFailed` (NO new variant); Display ≡ wrong-passphrase ≡ tampered at exit 4 (PIN-07). HKDF info enumeration test auto-discovers `cipherpost/v1/pin`. State-ledger schema migration: `state: Option<String>` field on `LedgerEntry` (v1.0 rows missing the field deserialize via serde default to None and map CONSERVATIVELY to `LedgerState::Accepted` — T-08-17, never silently classify Accepted as Burned). THREAT-MODEL.md §6.5 PIN mode + §6.6 Burn mode (multi-machine race, DHT-survives-TTL, indistinguishability, emit-before-mark). 309 tests pass / 0 failed / 19 ignored at Phase 8 close.

## Milestone Summary

**Decimal Phases:** None — straight Phases 5–9 sequence.

**Key Decisions Validated:**

- v1.1 milestone target = "Real v1" (PRD-closure + de-risking), NOT a launch milestone — held all the way through; "Three real users" / "Independent public review" criteria explicitly deferred to v1.2+
- Coarse phase granularity (5 phases) — every phase ended at user-visible capability; no phase was housekeeping-only
- Phase ordering: PASS+DOC → Phase 5; X509 → Phase 6 (pattern-establish); PGP+SSH → Phase 7 (pattern-apply); PIN+BURN → Phase 8 (compose over typed payloads); DHT → Phase 9 (network-class bugs deliberately last to not cascade back into freshly-shipped feature code)
- Traceability format locked project-wide (DOC-03): inline phase tags on each requirement are canonical; phase VERIFICATION.md files are authoritative for implementation status; no separate traceability table — eliminates the "Pending row" drift class that hit v1.0 (29 stale rows at close)
- Burn+receipt decision (BURN-04): receipt IS published after successful burn-receive; burn does not suppress attestation; receipt is delivery confirmation, not destruction proof
- PIN crypto = cclink-fork-with-divergence — Argon2id+HKDF→X25519→age preserves cipherpost's load-bearing "no direct chacha20poly1305" rule; cclink's direct AEAD path explicitly rejected during planning
- Burn emit-before-mark (D-P8-12) — emit-then-mark inverts v1.0's accepted-then-mark ordering for burn ONLY; safer failure mode (user keeps access on crash) compared to mark-then-emit (would lose user data to a half-completed state write)
- Single-retry-then-fail CAS contract on `publish_receipt` (D-P9-A1) — retry lives inside the trait method; final-conflict failures collapse into `Error::Transport`, no public `Error::CasConflict` variant (error-oracle hygiene preserved)
- Real-DHT triple-gate discipline (D-P9-D2 + Pitfall #29) — `#[cfg(feature = "real-dht-e2e")]` + `#[ignore]` + `#[serial]`; CI never runs `--features real-dht-e2e`; manual RELEASE-CHECKLIST is the only execution path
- pkarr defaults only — no `CIPHERPOST_DHT_BOOTSTRAP` env var in v1.1; bootstrap configurability deferred to v1.2+
