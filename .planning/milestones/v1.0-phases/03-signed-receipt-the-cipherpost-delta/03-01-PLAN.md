pub fn verify_record(record: &OuterRecord) -> Result<(), Error> {
    let pk = pkarr::PublicKey::try_from(record.pubkey.as_str()).map_err(|_| Error::SignatureInner)?;
    let vk = VerifyingKey::from_bytes(pk.as_bytes()).map_err(|_| Error::SignatureInner)?;
    let sig_bytes = base64::engine::general_purpose::STANDARD.decode(&record.signature).map_err(|_| Error::SignatureInner)?;
    let sig = Signature::from_slice(&sig_bytes).map_err(|_| Error::SignatureInner)?;
    let signable = OuterRecordSignable::from(record);
    let bytes = jcs(&signable)?;
    vk.verify_strict(&bytes, &sig).map_err(|_| Error::SignatureInner)?;
    let parsed: OuterRecordSignable = serde_json::from_slice(&bytes).map_err(|_| Error::SignatureCanonicalMismatch)?;

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DHT → cipherpost process | Any party can publish bytes under a pubkey they control; receipt body parse + sig verify must reject every non-recipient-signed or canonicalization-divergent variant before any field is surfaced |
| Rust attacker → serde round-trip | A tampered receipt whose JCS bytes parse-then-reserialize to something different is a canonicalization-bypass attempt (T-01-03-02 class) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Spoofing | `verify_receipt` (signature path) | mitigate | `VerifyingKey::from_bytes(recipient_pubkey)` + `verify_strict` on JCS bytes — an attacker cannot forge a receipt under a key they do not hold |
| T-03-01-02 | Tampering | `verify_receipt` step 5 (round-trip-reserialize) | mitigate | Parse-then-reserialize byte-compare; any canonicalization drift (extra whitespace, field-order variation, unicode normalization drift) fails with `Error::SignatureCanonicalMismatch` — mirror of `record.rs:138-143` T-01-03-02 defense |
| T-03-01-03 | Information Disclosure | Signature-failure error-oracle | mitigate | D-16 unified Display: all four sig-fail variants (`SignatureInner`, `SignatureCanonicalMismatch`, plus the outer two) share the string `"signature verification failed"`. Unit tests assert this invariant on every tampered-* test case |
| T-03-01-04 | Tampering | JCS canonical form drift between Rust versions | mitigate | Committed `tests/fixtures/receipt_signable.bin` — any toolchain change that shifts JCS bytes produces an immediate test failure with a clear message "past receipt signatures would no longer verify" |
| T-03-01-05 | Information Disclosure | Receipt fields (purpose, hashes, pubkeys) | accept | By design per D-RS-06 / CONTEXT.md "Receipts are fully public" — receipts are a social-layer attestation on a public substrate; no field is secret. Zeroize discipline does not apply |
| T-03-01-06 | Tampering | nonce entropy | mitigate | `OsRng.fill_bytes` reads `/dev/urandom` via `getrandom`; zero-init + 128 bits gives collision probability negligible at skeleton traffic scale (1–100 receipts/week/user per PITFALLS) |
