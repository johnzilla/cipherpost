**Badges (in this order):**

```markdown
[![CI](https://img.shields.io/github/actions/workflow/status/johnzilla/cipherpost/ci.yml?branch=main&label=CI&logo=github)](https://github.com/johnzilla/cipherpost/actions/workflows/ci.yml)

[![MSRV](https://img.shields.io/badge/rust-1.88%2B-orange.svg?logo=rust)](./rust-toolchain.toml)
[![Latest release](https://img.shields.io/github/v/tag/johnzilla/cipherpost?label=release&sort=semver)](https://github.com/johnzilla/cipherpost/tags)
```

Rationale per badge:
- **CI** — surfaces green/red at a glance; canonical for any project with GitHub Actions.
- **License: MIT** — standard hygiene; sourced from `Cargo.toml` `license = "MIT"`.
- **MSRV** — cipherpost has a hard MSRV pin (`rust-toolchain.toml`, CI gate); badge tells contributors before they `cargo build`.
- **Release tag** — last shipped milestone; cipherpost is not on crates.io (Cargo.toml
  version is `0.1.0` and pre-release), so a `crates.io` badge would be misleading.
  Using `v/tag` (not `v/release`) because the project tags milestones (`v1.0`, `v1.1.0`)
  but does not cut formal GitHub Releases.

**Verify:** `head -5 README.md` shows H1 + blank + badge row + blank + project blurb. No
"Status:" line remains. `grep -c '^\*\*Status:' README.md` returns `0`.

**Done when:** README.md edited; the four badge URLs and link targets are syntactically
