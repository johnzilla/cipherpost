## 4. Outstanding Human UAT

Both tests are flagged in their phase VERIFICATION.md files with `why_human:` explanations. Neither is a code gap — they prove the skeleton works against real infrastructure outside of mocks.

### UAT-1 — Phase 01: Interactive identity generation

**Test:** Run `cipherpost identity generate` interactively (no env var, real TTY).
**Expected:**
- Passphrase prompted twice (confirm).
- Identity file written at `~/.cipherpost/secret_key` mode `0600`.
- Both fingerprints (z32 + OpenSSH SHA256) printed on success.

**Why human:** Binary rejects piped stdin with `"TTY not available"` when stdout is not a terminal. Requires real TTY session.
**Priority:** Medium — code paths are unit-tested; this proves the UX end-to-end.

### UAT-2 — Phase 03: Real-DHT A→B→receipt round trip

**Test:** Two identities on different processes, send share via real Mainline DHT, recipient accepts, sender runs `cipherpost receipts --from <z32_b>` and sees a row matching the share.
**Expected:**
- Receipt table row with matching `share_ref`, `purpose`, and OpenSSH fingerprint.
- `--share-ref <ref>` returns the 10-field audit view.
- `--json` returns valid pretty-printed JSON.

**Why human:** MockTransport cannot reach Mainline DHT; this is a live-network smoke test.
**Priority:** High for shipping to real users, medium for archiving the skeleton — the automated E2E tests under MockTransport cover the logic; this proves the network binding works.
